================================================================================
                        ARCHITECTURE OVERVIEW
================================================================================

The CodeNova IDE frontend is a sophisticated Electron-based desktop application
that provides a VS Code-like interface for Java code refactoring. It consists of:

1. Main Process (Electron): Manages windows, file system, terminal, backend
2. Preload Script: Secure IPC bridge between render process and main process
3. Renderer Process: Monaco Editor, UI panels, real-time analysis
4. HTML/CSS: Workbench layout, styled tabs, activity bar, sidebar panels
5. Integration: FastAPI backend via IPC → HTTP proxy

Key Technologies:
- Electron.js (Desktop framework, 30 MB app)
- Monaco Editor (VS Code's editor, 70K lines omitted)
- xterm.js (Terminal emulation, 60K lines)
- Node-pty (Terminal spawning via C++ bindings)
- Electron-builder (Windows NSIS installer)
- Python FastAPI backend (packaged as subprocess)

ipcMain.handle('fs:readDir', async (event, dirPath) => {
  const entries = await fsPromises.readdir(dirPath, { withFileTypes: true });
  return entries
    .filter(e => !e.name.startsWith('.'))     // Hide dotfiles
    .map(entry => ({
      name: entry.name,
      path: path.join(dirPath, entry.name),
      isDirectory: entry.isDirectory(),
      isFile: entry.isFile()
    }))
    .sort((a, b) => {
      if (a.isDirectory && !b.isDirectory) return -1;  // Folders first
      if (!a.isDirectory && b.isDirectory) return 1;
      return a.name.localeCompare(b.name);              // Then alphabetical
    });
});
- **IPC HANDLER #3**: List directory contents (for file tree)
- withFileTypes: true = returns Dirent objects with isDirectory() method
- filter: skip dotfiles (.gitignore, .vscode, etc.)
- **SORTING PRIORITY**:
  1. Directories before files
  2. Within category, alphabetical order
- Returns array of {name, path, isDirectory, isFile}

ipcMain.handle('terminal:create', (event, { cols, rows, cwd }) => {
  if (!pty) throw new Error('node-pty is not available.');
  const shellPath = getDefaultShell();
  const id = nextTerminalId++;
  const term = pty.spawn(shellPath, getShellArgs(shellPath), {
    name: 'xterm-256color',
    cols: cols || 80,
    rows: rows || 24,
    cwd: cwd || os.homedir(),
    env: process.env
  });
  term.onData((data) => {
    if (mainWindow && !mainWindow.isDestroyed()) mainWindow.webContents.send('terminal:data', { id, data });
  });
  term.onExit(({ exitCode }) => {
    terminals.delete(id);
    if (mainWindow && !mainWindow.isDestroyed()) mainWindow.webContents.send('terminal:exit', { id, exitCode });
  });
  terminals.set(id, term);
  return id;
});
- **CRITICAL TERMINAL CREATION LOGIC**
- pty.spawn(): creates pseudo-terminal (PTY)
  - Like opening a shell in terminal where commands run
- arguments:
  - shellPath: /bin/bash on Unix, cmd.exe on Windows
  - getShellArgs(shellPath): --login on bash (initialize profile)
  - cols/rows: terminal width/height
  - cwd: current working directory (workspace folder)
  - env: inherit parent process environment (PATH, HOME, etc.)
- term.onData(): when shell outputs text
  - Send it back to renderer via IPC 'terminal:data'
- term.onExit(): when shell exits (user types exit or closes)
  - Send 'terminal:exit' with exit code
- **KEY STATE**: terminals.set(id, term)
  - Store this PTY in global Map so we can refer to it later
  - nextTerminalId++: each terminal gets unique ID

  const ext = path.extname(filePath).toLowerCase();
  const commands = {
    '.py': `python "${filePath}"`,
    '.js': `node "${filePath}"`,
    '.ts': `npx ts-node "${filePath}"`,
    '.rb': `ruby "${filePath}"`,
    '.go': `go run "${filePath}"`,
    '.sh': `bash "${filePath}"`,
    '.ps1': `powershell -File "${filePath}"`,
  };
  if (ext === '.java') {
    const cn = path.basename(filePath, '.java');
    const dir = path.dirname(filePath);
    if (JAVA_HOME) {
      const javacPath = path.join(JAVA_HOME, 'bin', 'javac');
      const javaPath = path.join(JAVA_HOME, 'bin', 'java');
      return { command: `cd "${dir}"; & "${javacPath}" "${path.basename(filePath)}"; if ($?) { & "${javaPath}" ${cn} }` };
    }
    return { command: `cd "${dir}"; javac "${path.basename(filePath)}"; if ($?) { java ${cn} }` };

  if (ext === '.c') return { command: `gcc "${filePath}" -o "${filePath}.exe"; if ($?) { & "${filePath}.exe" }` };
  if (commands[ext]) return { command: commands[ext] };
  throw new Error(`No run configuration for: ${ext}`);
});

SECTION 10: BACKEND API PROXY (Lines 579-620)

[CRITICAL LINE: Line 579-600]
async function backendFetch(endpoint, method, body) {
  const http = require('http');
  return new Promise((resolve, reject) => {
    const postData = body ? JSON.stringify(body) : null;
    const options = {
      hostname: '127.0.0.1',
      port: BACKEND_PORT,
      path: endpoint,
      method: method,
      headers: {
        'Content-Type': 'application/json',
        ...(postData ? { 'Content-Length': Buffer.byteLength(postData) } : {})
      },
      timeout: 30000
    };
    const req = http.request(options, (res) => {
      let data = '';
      res.on('data', chunk => data += chunk);
      res.on('end', () => {
        try { resolve(JSON.parse(data)); }
        catch { resolve(data); }
      });
    });
    req.on('error', reject);
    req.on('timeout', () => { req.destroy(); reject(new Error('Backend request timed out')); });
    if (postData) req.write(postData);
    req.end();
  });
}
- **THE CRITICAL HTTP BRIDGE** (IPC → REST API)
- Why this exists: Renderer can't make HTTP requests (sandboxed)
- Instead, renderer sends IPC → main.js → main.js makes HTTP
- Builds HTTP request:
  - hostname: 127.0.0.1 (localhost)
  - port: 8000 (our FastAPI backend)
  - path: /refactor, /check-errors, /git/commit, etc.
  - method: GET or POST
  - headers: Content-Type: application/json
- Promise-based: async/await compatible
- Error handling:
  - Network error → reject promise
  - Timeout 30s → destroy connection, reject
  - Response → parse JSON (or raw string if not JSON)

SECTION 2: SECURITY MODEL

The preload.js enforces security through:
1. Whitelist pattern: only exposed methods can be called
2. No require(): renderer can't dynamically load modules
3. No global access: window.fs, window.require don't exist
4. Message passing: all operations go through IPC (logged/filtered)
5. No direct file access: all FS ops must go through main.js

Example attack mitigated:
  Malicious script in renderer: window.fs.readFile('/etc/passwd')
  Result: fs not exposed → TypeError: window.fs is undefined ✓

Real interface:
  Legitimate code: await api.readFile('./MyFile.java')
  Result: IPC call → main.js → fs.readFile() → response ✓

function createTab(filePath, content, isUntitled = false) {
  const existing = state.openTabs.find(t => t.filePath === filePath && !t.isUntitled);
  if (existing) {
    activateTab(existing.id);

  return tab;
}

function drawRadarChart(canvas, radar) {
  if (!radar || !radar.labels) return;
  const ctx = canvas.getContext('2d');
  const rect = canvas.parentElement.getBoundingClientRect();
  canvas.width = rect.width * (window.devicePixelRatio || 1);
  canvas.height = rect.height * (window.devicePixelRatio || 1);
  ctx.scale(window.devicePixelRatio || 1, window.devicePixelRatio || 1);

  const w = rect.width;
  const h = rect.height;
  const cx = w / 2;
  const cy = h / 2;
  const r = Math.min(cx, cy) - 30;
  const labels = radar.labels;
  const values = radar.values;
  const n = labels.length;
  const angleStep = (2 * Math.PI) / n;
  const startAngle = -Math.PI / 2;

  ctx.clearRect(0, 0, w, h);

  // Draw grid rings
  for (let ring = 1; ring <= 5; ring++) {
    const ringR = (r / 5) * ring;
    ctx.beginPath();
    for (let i = 0; i <= n; i++) {
      const angle = startAngle + i * angleStep;
      const x = cx + ringR * Math.cos(angle);
      const y = cy + ringR * Math.sin(angle);
      i === 0 ? ctx.moveTo(x, y) : ctx.lineTo(x, y);
    }
    ctx.closePath();
    ctx.strokeStyle = 'rgba(255,255,255,0.08)';
    ctx.lineWidth = 1;
    ctx.stroke();
  }

  // Draw axes...
  // Draw data polygon...
  // Draw data points...
  // Draw labels...
  // Draw value labels on points...
}

================================================================================
                        CRITICAL IPC FLOW EXAMPLES
================================================================================

To understand how frontend and backend communicate, here are key flows:

FLOW 1: File Save
  1. User types in Monaco editor
  2. User presses Ctrl+S
  3. renderer.js receives keyboard event → calls saveFile(tabId)
  4. Gets content from tab.model.getValue()
  5. Calls api.writeFile(filePath, content)
  6. IPC: ipcRenderer.invoke('fs:writeFile', filePath, content)
  7. main.js receives IPC → ipcMain.handle('fs:writeFile', ...)
  8. main.js calls fsPromises.writeFile(filePath, content)
  9. File written to disk
  10. Promise resolved, returned to renderer
  11. Renderer marks tab as not dirty (removes dot)

FLOW 2: Refactoring Code
  1. User clicks "Refactor Code" button in Refactoring panel
  2. Calls refactorCode() in renderer.js
  3. Gets selected refactoring types from checkboxes
  4. Calls api.backendRefactorReview({code, types})
  5. IPC: ipcRenderer.invoke('backend:refactorReview', ...)
  6. main.js receives → calls backendFetch('/refactor/review', 'POST', body)
  7. backendFetch() makes HTTP call to http://127.0.0.1:8000/refactor/review
  8. Python FastAPI backend receives request
  9. Runs JavaRefactoringEngine.refactor(code, types)
  10. Returns refactored_code, diff, metrics_before, metrics_after, session_id
  11. HTTP response returned to backendFetch()
  12. Promise resolved in renderer
  13. Renderer shows diff overlay with side-by-side comparison
  14. User clicks "Accept"
  15. Calls api.backendRefactorDecision({session_id, action: 'accept'})
  16. Backend returns final_code
  17. Renderer updates tab.model with new code

FLOW 3: Terminal Input/Output
  1. User types 'ls' in terminal and presses Enter
  2. xterm.onData fires with data = 'ls\r'
  3. Calls api.writeTerminal(terminalId, 'ls\r')
  4. IPC: ipcRenderer.send('terminal:write', {id, data})
  5. main.js receives → terminal.write('ls\r')
  6. Sends command to PTY process (running bash shell)
  7. Shell executes 'ls' and outputs results
  8. term.onData in main.js fires with output
  9. main.js sends IPC: mainWindow.webContents.send('terminal:data', {id, data: output})
  10. renderer.js api.onTerminalData fires
  11. Calls term.write(output)
  12. xterm displays output on screen
  13. User sees: $ ls\r\nfile1.txt file2.txt\r\n$

================================================================================
                          FRONTEND ARCHITECTURE SUMMARY
================================================================================

The CodeNova IDE frontend follows a 5-layer architecture:

LAYER 1 - SECURITY BOUNDARY (preload.js):
- Exposes only 100+ safe API methods via contextBridge
- Prevents renderer from accessing dangerous Node APIs
- All IPC requests validated at entry point

LAYER 2 - MAIN PROCESS (main.js):
- Manages window lifecycle and OS integration
- Spawns Python FastAPI backend subprocess
- Proxies HTTP requests from renderer to backend
- Handles file system, terminal, Git operations
- 900 lines managing critical desktop operations

LAYER 3 - RENDERER PROCESS (renderer.js):
- React-like state management (global state object)
- Monaco editor integration (syntax highlighting, tabs)
- Terminal integration (xterm.js + pty bidirectional)
- Refactoring workflows (preview, diff, accept/reject)
- Real-time error checking with debouncing
- 2500 lines of complex UI logic

LAYER 4 - HTML/STYLING (index.html + styles.css):
- Workbench layout: activity bar | sidebar | editor | panels
- Flexbox responsive design (resizable sidebar)
- VS Code Dark+ theme colors
- Component styling (tabs, tree, terminal, modals)

LAYER 5 - BACKEND API (FastAPI Python):
- Java refactoring engine (AST analysis)
- Real-time error detection
- Git operations (GitPython)
- Code metrics (Halstead, cyclomatic complexity)
- Multi-file dependency analysis
- AI chat endpoint (OpenAI GPT-4o-mini)

DATA FLOWS:
- User actions in renderer → IPC → main process → HTTP → FastAPI
- File drag-drop → file explorer → renderer state update → UI re-render
- Terminal input → xterm.js → IPC → PTY in main → shell output → IPC → xterm
- Refactoring: user click → preview dialog → accept/reject decision → apply to editor

PERFORMANCE OPTIMIZATIONS:
- Lazy-load file tree (don't load entire project tree on startup)
- Debounced error checking (800ms after user stops typing)
- Debounced graph updates (update on tab switch, not every keystroke)
- Terminal FitAddon auto-resize (ResizeObserver, not manual)
- Monaco DiffEditor disposed and recreated (prevent memory leaks)

SECURITY:
- Context isolation: renderer can't access Main process context
- Process sandboxing: renderer can't require('fs') directly
- IPC whitelisting: only exposed API available to renderer
- Content Security Policy: restrict external requests
- JAVA_HOME detection: prevent RCE via malicious Java version

2. Click file → creates tab, Monaco loads with syntax highlighting
3. Type code → status bar updates line/col, debouncer schedules error check after 800ms
4. Press F5 → run file compiles Java and shows output in terminal
5. Click Refactor → sends code to FastAPI, shows diff preview
6. Click Accept → applies refactored code to tab, saves history
7. View terminal → bidirectional input/output between xterm and PTY shell
