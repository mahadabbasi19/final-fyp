**CRITICAL LINE**: Line 20-43 - Mathematical formulation of Halstead Complexity

Halstead Measures (Developed by Maurice Halstead 1977):
- η1 (eta1) = number of UNIQUE OPERATORS (if, for, +, -, etc.)
- η2 (eta2) = number of UNIQUE OPERANDS (variable names, literals)
- N1 = total count of all operators
- N2 = total count of all operands

**CRITICAL FORMULAS** (documented in comments):
Line 35: Volume V = N × log₂(η) 
  - Measures "information content" or "size" expressed in bits
  - Bigger = more complex program

Line 36: Difficulty D = (η1 / 2) × (N2 / η2)
  - Measures error-proneness
  - High D = likely to have bugs

Line 37: Effort E = D × V
  - How much mental effort programer needs
  - E / 3600 = estimated hours to understand code

Lines 472-580: ConditionSimplifier class

**CRITICAL REFACTORING #3: Simplify Boolean Conditions Using De Morgan's Laws**

De Morgan's Laws (Boolean algebra):
  NOT (A AND B) = (NOT A) OR (NOT B)
  NOT (A OR B) = (NOT A) AND (NOT B)

This refactoring applies these laws automatically.

Example:
  Original: if (!(x > 5 && y < 10)) { ... }
  Simplified: if (!(x > 5) || !(y < 10)) { ... }
  Even better: if (x <= 5 || y >= 10) { ... }

**CRITICAL LINE 494-510**: analyze_conditionals() method
- **CRITICAL LINE 500-510**:
  m = re.search(r'if\s*\(\s*!\s*\((.+?)\)\s*\)', stripped)
  Matches: if (!(condition))
  Extract inner condition: condition
  If condition has && or ||:
      Opportunity to apply De Morgan's Law
- **CRITICAL LINE 512-522**:
  Detect guard clauses (early returns to reduce nesting)
  if depth >= 3:
      Report opportunity to use guard clause

Example of guard clause:
  Before:
    if (user != null) {
        if (user.isAdmin()) {
            if (user.hasPermission()) {
                processRequest();
            }
        }
    }
  
  After (guard clause):
    if (user == null) return;
    if (!user.isAdmin()) return;
    if (!user.hasPermission()) return;
    processRequest();
  
  Much more readable! Nesting reduced from 3 to 0.

**CRITICAL LINE 525-545**: simplify() method
- Applies all deterministic condition simplifications
- For each matched pattern, calculates simplified version
- Creates RefactoringAction with original and simplified code

================================================================================
                       CRITICAL CONCEPTS FOR TEACHERS
================================================================================

1. CYCLOMATIC COMPLEXITY (cc)
   - Measures decision paths in code
   - cc = 1 + number of (if, for, while, switch cases)
   - Why: each path = must test separately
   - cc 1-5: simple, easy test
   - cc 5-10: moderate, needs refactoring
   - cc 10+: complex, MUST refactor (decompose behavior)

2. HALSTEAD METRICS (Program difficulty)
   - η1 = unique operators, η2 = unique operands
   - Difficulty D = (η1/2) × (N2/η2)
   - Higher D = more error-prone code
   - Helps predict bugs before they happen

3. MAINTAINABILITY INDEX (MI)
   - Score 0-100 (higher is better)
   - Combines: Halstead Volume + Cyclomatic Complexity + LOC + Comments
   - MI > 85: green (good)
   - 65 < MI < 85: yellow (needs work)
   - MI < 65: red (critical, refactor immediately)

4. BEHAVIOR PRESERVATION PROTOCOL
   - CRITICAL FOR SAFETY OF REFACTORING
   - pre_check(): original code must parse, no errors
   - post_check(): refactored code must parse, NO NEW ERRORS
   - safe_apply(): if post_check passes, apply. Otherwise roll back.
   - This is why CodeNova refactorings are SAFE (verified correct)

5. VARIABLE SCOPE ANALYSIS
   - When extracting code slice to new method:
     - in_params = variables used but not declared in slice
     - out_params = variables modified in slice and used after
     - Determine: void/single_return/data_class resolution
   - This ensures extracted method signature is correct

6. INTENT ROUTING IN CHAT
   - Don't call LLM for refactoring (use deterministic engine)
   - Don't call LLM for metrics (calculate locally)
   - Only call LLM for creative tasks (generation, advice)
   - Saves cost, ensures safety, improves speed

7. DEBOUNCING FOR IDE PERFORMANCE
   - Without: 100-character code = 100 error checks = CPU spike
   - With debouncing: 100-character code = 1 error check (after pause)
   - Uses threading.Timer + locking for thread-safety
   - Keeps UI responsive while user types

8. REST API CONTRACTS (Pydantic models)
   - Frontend and Backend must share understanding of data format
   - Pydantic validates ALL incoming requests
   - If format wrong, returns 422 validation error
   - Prevents bugs where frontend sends {"code": "..."} but backend expects {"java_code": "..."}

9. MULTI-FILE DEPENDENCY ANALYSIS
   - For each file: parse, extract classes, extends, implements
   - Cross-file: regex to find type references (new ClassName, ClassName.method, etc.)
   - Build nodes (files) and edges (dependencies)
   - Visualize as graph (shows coupling/architecture)
